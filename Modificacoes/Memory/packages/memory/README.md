# CyberVinci Memory

Memory is a native, local-first CyberVinci extension for project context discovery. It does not install or require `code-review-graph`, `agentmemory`, Python, Docker, Redis, Qdrant, Postgres, or an external service.

## Final architecture

Memory remains a native Theia/CyberVinci extension, split between the browser UI, backend service, shared contracts, local analyzers, repositories, retrieval sources, and MCP contribution in this package. SQLite is the preferred local source of truth behind `MemoryStoreRepository`; the legacy JSON store remains a migration, import/export, and emergency fallback path. The extension reads workspace files only through controlled indexing and does not create hidden learning or indexing state inside a project unless the user explicitly installs a portable package under `.cybervinci/memory`.

The architecture is intentionally local-first. Code graph extraction, memory lifecycle, Knowledge Graph derivation, Context Cart ranking, feedback, benchmark reports, audit events, and portable artifacts are produced from local data by default. Optional vector search, transcript search, document/media ingestion, remote image semantics, remote media transcription, IDE/global memory export, and portable package installation are all opt-in settings or explicit commands. Enabling an option is consent to use an already configured capability; Memory must not silently download models, start containers, call paid APIs, or upload repository data.

Graphify is a compatibility target and design inspiration only. Memory can emit Graphify-shaped graph JSON and import `graphify-out/graph.json` as review candidates, but Graphify is not a runtime dependency, service dependency, required CLI, storage backend, or replacement for the native Memory model.

## Knowledge model

Memory records and memory-derived candidates use five scopes:

| Scope | Locator | Retrieval behavior |
| --- | --- | --- |
| `global` | No `workspacePath`; IDE-wide memory. | Visible across workspaces and omitted from workspace exports unless explicitly included. |
| `workspace` | `workspacePath`. | Bound to the opened local workspace. |
| `repository` | `repositoryUrl` and/or normalized `repositoryId`. | Shared between clones of the same remote; local path is not required for reuse. |
| `session` | `sessionId`, usually with `expiresAt`. | Temporary; expired records are excluded from normal retrieval and export. |
| `task` | `taskId`, `retentionPolicy`, usually with `expiresAt`. | Temporary/task-bound; searched only with matching task context unless explicitly requested. |

Memory Spaces are persisted entities used by the backend and UI to group visible memories by IDE, workspace, repository, session, and task scope. They provide counters by scope/status and prevent the frontend from inferring spaces from a combined memory list. Imported portable memories, deterministic contradictions, document-derived memories, Knowledge Graph imports, and consolidation output are review candidates by default and must not become `active` automatically unless they are direct manual user memories or temporary in-session records.

The normalized Knowledge Graph is stored as graph metadata plus concept and link rows, with fallback reconstruction to the compatible `MemoryGraph` shape. Concepts and links carry source, evidence, confidence, origin markers, status, scope/repository locators, and review metadata where applicable. Surprising connections and imported graph content are candidates until reviewed, so normal retrieval and Context Cart suggestions only use accepted graph knowledge.

## Opt-in and privacy matrix

Memory is disabled on first workspace open. A workspace is eligible for indexing, event capture, retrieval learning, or memory writes only after the user explicitly enables Memory for that workspace. Capability toggles are independent: enabling the base workspace setting does not imply consent for sensitive capture, prompt snippets, optional document/media ingestion, vector search, remote analysis, exports, or automatic chat learning.

The following matrix is the consent contract for workspace settings and UI controls:

| Capability | Default | Consent control | Captured or persisted data | Privacy rule |
| --- | --- | --- | --- | --- |
| Workspace Memory | Off | `enabled` for the current workspace | Workspace identity, repo locator, branch/head metadata, settings timestamps, and safe status metadata | No indexing, learning, retrieval cache, or event capture starts until this is enabled. |
| Code graph | Off | `graphEnabled` and `optIn.codeGraph` | File metadata, symbols, code chunks, relations, graph snapshots, change impact, and analyzer status | Reads only allowed workspace files; ignores denylisted, generated, binary, large, ignored, and secret-bearing content before chunking. |
| Project/repository memory | Off | `memoryEnabled`, `optIn.projectMemory`, and review actions | User-approved memories, memory candidates, scope/repository locators, evidence markers, lifecycle metadata, and access counters | Generated/imported memories are candidates by default; sensitive memory content is flagged for review and is not embedded. |
| Chat learning | Off | `chatLearningEnabled` | Prompt signature, normalized intent/language/target/action, prompt hash, timestamps, source, and redaction metadata | Raw prompts are not persisted by default. Learning events use minimized hashes and metadata unless prompt snippet consent is enabled. |
| Redacted prompt snippets | Off | `optIn.promptSnippets` | Short redacted prompt samples for review, skill lifecycle, or diagnostics | Requires separate opt-in from chat learning; snippets pass through secret scanning and redaction before persistence. |
| Skill suggestions | Off | `skillSuggestionsEnabled` and `optIn.skills` | Skill candidate signature, repeated-pattern counters, status, rejection counts, redacted examples when separately allowed, and lifecycle timestamps | Suggestions are based on normalized repeated intent, not full raw prompt fingerprints; repeated rejections silence candidates. |
| Context Cart | Off | `optIn.contextCart` plus explicit item acceptance | Suggested context item IDs, source kind, token estimate, ranking explanation, acceptance/rejection feedback, and approved pack metadata | Context is never injected into AI Chat until the user accepts items and runs Insert Approved Context; audit events store IDs/counts, not raw context. |
| Hover/editor hints | Off | `editorHoverEnabled` and `optIn.editorHover` | Editor lookup source, symbol IDs, relation IDs, and local hint metadata | Hints are local and read from accepted graph/memory state; they do not send editor content to a model. |
| Event capture bus | Off | `optIn.events`, gated by base workspace enablement | Sanitized events for prompts, tools, files, searches, terminal, build/test, context, memory, skills, and agent activity | Event payloads use IDs, hashes, counts, scopes, statuses, paths, and boolean evidence markers; secrets and raw prompts are excluded by default. |
| Transcript search | Off | `optIn.transcriptSearch` | Redacted transcript session/message metadata and searchable snippets | Recording/search is local and scoped by retention policy; secret-bearing messages are blocked or redacted before persistence. |
| Local sensitive documents | Off by format | `optIn.pdfDocuments`, `optIn.officeDocuments`, `optIn.images`, `optIn.diagrams`, and `optIn.audioVideo` | Redacted extracted text, safe media metadata, document chunks, and review-only document memory candidates | Optional documents and media are not read unless their format is enabled; extraction is local and redacted before storage. |
| External/local docs collections | Off | `optIn.externalDocCollections` and per-collection policy | Collection identity, safe index metadata, retrieved snippets, and ranking metadata | Installed library packages remain global IDE assets; workspace association and retrieval require explicit policy. |
| Vector search | Off | `optIn.vectorSearch` | Local vector rows for eligible, non-sensitive memories or chunks when a backend is already configured | Memory does not download models, start services, call paid APIs, or embed sensitive content silently. |
| Remote image/media semantics | Off | `optIn.remoteImageSemantics`, `remoteImageSemanticsConsentAt`, `optIn.remoteMediaTranscription`, and `remoteMediaTranscriptionConsentAt` | Consent timestamp and provider-result metadata when the host application explicitly configures a provider | The built-in indexer does not call remote providers; remote processing requires recorded consent and host configuration. |
| IDE/global memory export | Off | `exportOptions.includeGlobalMemories` or explicit export option | Selected IDE/global memories and sanitized portable metadata | Workspace exports omit IDE/global memories unless the caller explicitly includes them. |
| Portable package install | Off | Explicit UI/MCP install confirmation | Redacted report, graph, context, policy, and optional sanitized memory artifacts under `.cybervinci/memory` | Detection reads safe metadata only; install/import never activates memories or graph knowledge automatically. |

These defaults are part of the feature contract. New Memory capture points must first check the workspace enablement state and the relevant capability consent, then persist the minimum viable metadata. If a feature needs content beyond hashes, IDs, counts, redacted snippets, or source metadata, it must add a separate opt-in and document that storage here.

## Retrieval storage

Local documentation retrieval delegates to `LibraryService.searchDocs` from `@cybervinci/library`, which owns versioned library docs, SQLite FTS5 creation, BM25 search, and JSON fallback. Memory does not create a second documentation index.

Code chunks are still Memory data and are part of the Phase 1 SQLite source-of-truth contract for dashboards, export/import, and fallback search. For retrieval, the repository indexes those chunks through `LibraryService.indexWorkspaceSections` with source `memory` and searches them through `LibraryService.searchWorkspaceSections`. The library store keeps these generic workspace sections separate from installed documentation packages under `~/.cybervinci/library/workspace-sections`, backed by `node:sqlite` FTS5 when available and JSON when it is not. If an older library service does not expose the workspace-section API, Memory falls back to the previous local `code-chunks.db`/JSON search path under `~/.cybervinci/memory`.

Workspace indexing is incremental by default. `indexWorkspace` compares the current file metadata with the previous store state, preserves chunks for files that did not change, and chunks only changed files unless the caller explicitly requests a broader backfill. Callers can pass `scope: 'local-docs'` to index only local documentation/configuration text (`.md`, `.txt`, `.json`, `.yaml`, `.yml`) plus explicitly opted-in PDF/Office/audio/video documents before attempting full code coverage, and `maxFiles` to cap a controlled backfill batch. Audio/video ingestion is off by default: when `optIn.audioVideo` is enabled, the local indexer only reads conventional sidecar transcripts (`.transcript.txt`, `.txt`, `.vtt`, `.srt`) next to the media file or stores safe media metadata. Transcript chunks always include transcription origin and confidence, are redacted before chunking, and can propose review-only memory candidates. `optIn.remoteMediaTranscription` requires `remoteMediaTranscriptionConsentAt`; the built-in local indexer records that consent in metadata but does not call a remote transcription provider. Memory vector backfill is never triggered by ordinary indexing; callers must pass `backfillMemories: true`, and the existing vector-search enablement plus user consent checks still apply.

External documentation collections are workspace policies under `externalDocCollections` and are disabled unless `optIn.externalDocCollections` is true. They are refreshed only when `indexWorkspace` is called with `refreshExternalDocs: true` (optionally narrowed by `externalDocCollectionIds`), store traceable source/origin/collection metadata on `external-docs` chunks, redact content before chunking, and never scan arbitrary external roots during ordinary workspace indexing.

Project memory retrieval remains compatible with JSON and in-memory repositories. Repositories that expose the optional `searchMemoryItems(workspacePath, ftsQuery, limit)` method are used by the retrieval broker for `project-memory`, `repository-memory`, and `task-memory` queries. The broker passes a sanitized FTS `MATCH` expression, accepts repository-provided BM25-derived scores/snippets/evidence, filters `project-memory` results to global memories plus the current workspace/session scope, filters `repository-memory` by the current `repositoryUrl` or `repositoryId`, and exposes task/ephemeral memories only through `task-memory` when the query carries the matching `taskId`, a retention policy is present, and `expiresAt` is still valid. Expired `session`/`task` memories are dropped before ranking, and the broker falls back to the JSON scan path if the optional method is unavailable or fails. Workspaces without a Git remote use the deterministic local `repositoryId` fallback, so repository-scoped memories created there are retrievable without requiring Git.

Context suggestions use a configurable local ranking formula when retrieval results provide structured `rankingSignals`. The default weights are `0.30 * bm25Score + 0.25 * vectorScore + 0.15 * graphScore + 0.10 * memoryWeight + 0.10 * recencyScore + 0.05 * acceptanceScore + 0.05 * scopeBoost - stalePenalty`, with an explicit local `feedbackMultiplier` applied afterward when unresolved feedback matches the result. Code-graph Context Cart results fold local god-node, community, surprising-connection, and change-risk analyses into `graphScore` and expose the contributing sub-signals in the ranking explanation. Callers may override weights through `rankingWeights` on the suggestion request and on `cybervinci.context.explainRanking`. Results without structured signals keep their existing `score` behavior for JSON fallback compatibility. Suggested context is still clipped by the token budget after ranking, and suggestion reasons include source evidence plus scope, stale status, importance, weight, graph signals, and feedback multiplier when available.

Retrieval results are cached locally in `pi_retrieval_cache` as one compact record per normalized query, scope parameters, enabled sources, limit, repository identity, task identity, and source-data hash. Cache entries expire after five minutes and are reused only when the hash of retrieval source tables still matches; store writes clear the table, so changed memories, code chunks, graph rows, events, skills, or feedback invalidate stale cache data. The JSON fallback stores the same compact records in `retrieval-cache.json` next to the fallback store.

Feedback records are a first-class retrieval source for Context Cart under `feedback-record`. Only unresolved records from the active workspace are suggested, and snippets include the feedback outcome, target source/id, reason, review correction, and evidence so stale reports and rejected corrections can be reviewed without activating or rewriting the original memory/context item.

Benchmark datasets are versioned in code under `benchmark-datasets` and exposed through `BenchmarkService.listVersionedSuites()`, `getVersionedSuite(id, version)`, and `validateSuite(suite)`. The built-in `memory-core-v1` suite covers retrieval, explainable ranking, security filtering, indexing fallback, and multi-session/task memory behavior across global, workspace, repository, session, and task memory with deterministic expected ids/terms and minimum recall. Benchmark reports include Context Cart rejection rate, accepted/rejected suggestion counts, unresolved feedback count, and feedback multiplier usage in ranking, split into boosted and penalized result counts. Workspace indexing also returns a local indexing-latency benchmark with file/chunk throughput and metadata-only counts for ignored, sensitive, large, generated, binary, and multi-language files. Benchmark reports are persisted locally as workspace history in JSON fallback storage and the `pi_benchmark_reports` SQLite table, and the Memory panel shows the latest report plus recent local runs. Benchmark reports remain local, redacted, and source-metadata only: sensitive files are counted as security expectations but raw secret content is never included in cases or exported reports.

## Optional vector search

Vector search is an optional, local-only retrieval enhancement. It is off by default and is controlled by the Settings tab through the `optIn.vectorSearch` setting. Enabling the setting is only consent for Memory to use a compatible vector backend that is already installed and configured; Memory must not silently download embedding models, start external services, or send workspace content to a remote provider. The built-in SQLite store exposes a local vector-search backend over the persisted `pi_memory_vectors` table; builds may replace that implementation with `sqlite-vec` when the extension is locally available, but vector retrieval remains disabled until the workspace setting is enabled and consent is recorded.

Before any memory text is embedded, Memory runs the deterministic `SecretScanner` over the exact memory content that would be vectorized. Memories with secret-like values are skipped during vector backfill and any existing vector for that memory is removed when an update introduces sensitive content. The shared vector service also refuses direct embedding requests for code chunks, local documentation, memories, and transcript/audit payloads when the source is marked sensitive or the text contains a secret-like value. They remain available for human review through the normal lifecycle, but sensitive content is never written into embedding vectors or `pi_memory_vectors`.

The Settings tab reports the local model label and backend status when a compatible backend exposes a vector status API. If no backend is installed, the UI explicitly reports that BM25 and local FTS retrieval remain active. Manual vector backfill is shown only as a wait-compatible optional API call (`backfillVectorSearch`, `rebuildVectorSearch`, or `rebuildVectorIndex`) and remains disabled on builds that do not expose one.

BM25/FTS retrieval remains the guaranteed fallback for code chunks, workspace sections, local docs, and project memories. Optional vector results are ranked by their reported score alongside BM25 results, so a disabled, empty, unavailable, or failing vector backend does not suppress lexical matches.

## Knowledge Graph UI and MCP

The Knowledge Graph tab reads `dashboard.graphs.knowledge` when a compatible backend supplies it. If the graph is not present, the frontend probes an optional `getKnowledgeGraph` service method and otherwise renders a memory-derived concept graph from existing Memory records. The tab shows concept and link lists next to the graph and can export the visible graph as JSON, Markdown, or Graphviz DOT. Export uses an optional `exportKnowledgeGraph` service method when available and falls back to local serialization.

Surprising connections are deterministic Knowledge Graph link candidates between distant concept areas that share meaningful terms. They are always stored as `candidate` links with `linkKind: surprising_connection`, `classification: inferred`, and `reviewRequired: true`; they do not become active without review.

MCP exposes `cybervinci.knowledge.search`, `cybervinci.knowledge.inspect`, `cybervinci.knowledge.export`, `cybervinci.knowledge.create`, `cybervinci.knowledge.add`, and `cybervinci.knowledge.link`. Search, inspect, and export remain read-only and use dashboard fallback data when the knowledge backend is not installed. Create, add, and link are wait-compatible with future writable backend APIs and return an explicit unsupported result on older builds.

## MCP surface

The MCP contribution is a compact, token-budget-aware surface over the same backend service used by the UI. It exposes context suggestion and pack creation, ranking explanation, graph queries and pull-request analysis, memory search/lifecycle/scope operations, Memory Spaces, Knowledge Graph read/write-compatible operations, feedback record/search/resolve operations, transcript session/message/search/forget operations, and portable package export/compare/install operations. MCP tools return source, scope, evidence, stale status, importance/weight when available, feedback impact, and ranking reason for suggested context instead of returning opaque scores.

All MCP operations preserve the local-first safety model. Transcript search requires workspace opt-in, portable package install writes only under `.cybervinci/memory`, compare operations do not import or activate memories, and fallback implementations report unsupported or use sanitized audit data instead of failing with raw internal state.

## Portable package and Graphify compatibility

Portable workspace intelligence is always explicit. The UI action `Install portable intelligence in project` and the MCP tool `cybervinci.graph.installPortablePackage` require confirmation before writing redacted artifacts to `.cybervinci/memory`. Detection of an existing package reads safe metadata only and never imports memories, indexes artifact content, or activates candidates automatically.

The portable package may include:

- `cybervinci-project-report.md` with redacted architecture, module, risk, decision, convention, suggested-question, and context-suggestion summaries.
- `cybervinci-project-graph.json` with safe nodes, edges, concepts, links, confidence, evidence markers, origin markers, and classification metadata.
- `cybervinci-project-graph.graphml` for external graph tooling.
- `cybervinci-project-graph.html` as a standalone local graph viewer with no network script or IDE runtime dependency.
- `cybervinci-project-callflow.svg` with redacted dependency/call-flow labels.
- `graphify-out/graph.json` as a Graphify-compatible nodes/edges export.
- Optional sanitized context pack, memories, Knowledge Graph candidates, metadata, and policy information within the export limits.

Exports omit IDE/global memories by default, omit expired session/task memory, and sanitize repository-scoped memory so clone-local paths are not written into portable records. Imports and Graphify JSON ingestion compute a diff first and store safe new content as review candidates with portable/import origin markers; they never make imported memories, concepts, or links active without human review.

## Optional ICM bridge

The ICM bridge is a data-format bridge only. Memory can import ICM-style memories as `MemoryItem` records and map ICM memoir concepts/links into `KnowledgeGraph` records. Export produces an ICM-style JSON bundle with memories, memoirs, and explicit limits.

Imported ICM memories are stored as `candidate` records by default so the IDE approval workflow stays in control. Callers can opt into immediate activation with the import request, but ICM is never required for Memory to function. The Settings tab stores an optional ICM binary path for future integrations; current import/export paths do not execute that binary, download anything, or send workspace data outside the machine.

## Memory lifecycle UI

The Memory Health tab is a lifecycle dashboard backed by `getDashboard`. It reads existing memory records, groups them by stale state and scope, shows low-weight and never-used counters, and lists recent consolidation audit events without raw memory content. The shared `memoryHealth` contract also returns compact issue counters and issue entries for duplicates, contradictions, source-less active memories, sensitive content, misplaced global scope, old candidates, low-weight memories, and never-used memories, so reviewers can triage risks before archiving/removing anything. On branches that already return `importance`, `weight`, `accessCount`, or last-access timestamps on memory items, those values appear in Memory Health, IDE Memories, and Project Memories.

`getDashboard` includes `memorySpaces`, the persisted spaces visible to the current workspace. The IDE Memories, Project Memories, Repository Memories, Session Memory, and Task Memory tabs render those spaces directly instead of relying only on a combined frontend scope filter, so reviewers can see whether a memory is tied to the IDE, the local project, a shared repository identity, the current session, or a retained task.

The current shared memory contract includes `staleStatus`, so the UI can mark memories as `fresh`, `possibly_stale`, or `stale` through the existing `updateMemory` RPC. Explicit usage should be recorded through `updateMemoryAccess`, which increments `accessCount`, refreshes `lastAccessedAt`, and applies the bounded lifecycle weight boost. Importance actions are wait-compatible: they appear only when a memory item already has an `importance` property, then send the optional field through the same update path for compatible backends. Project and Repository memory rows can be promoted to IDE/global memory through `promoteMemoryToIde`; IDE/global rows can be demoted back to the current Project/workspace through `demoteMemoryToWorkspace`. Both actions ask for confirmation and preserve evidence while the backend writes the sanitized audit event.

The Memory Candidates tab is the approval queue for records whose memory `status` is `candidate`. Reviewers can edit title/content inline, approve a candidate into `active`, reject it, or archive it through the existing `updateMemory` RPC. IDE Memories and Project Memories exclude candidate records so proposed memories stay in the review queue until a user decides.

Topic consolidation is review-first. The memory scope tabs expose a Consolidate action when at least two records are visible; it asks for a topic and calls `proposeMemoryConsolidation` with those memory IDs. The backend creates a `candidate` memory with `source: memory-consolidation` and `supersedes` links to related active memories, skipping any source memory whose title/content matches the secret scanner. The related memories are not changed while the consolidation is only a candidate; when a reviewer approves that candidate through `updateMemory({ status: 'active' })`, the original memories are marked with `supersededBy` pointing at the approved consolidated memory.

The Feedback tab uses `searchFeedback` when available and otherwise falls back to sanitized audit events. Unresolved backend feedback records expose a Resolve action that calls `resolveFeedback({ workspacePath, id })`, refreshes the feedback list, and leaves the original memory/context item unchanged for separate human review.

AI Chat memory learning also feeds that queue. If a backend exposes a compatible optional `extractMemoryCandidates` RPC, the frontend calls it after deterministic chat analysis and persists returned title/content proposals as `candidate` memories when needed. LLM-assisted memory proposals are stored as candidates instead of immediately active memories, using optional candidate RPCs when present and falling back to `addMemory` followed by `updateMemory({ status: 'candidate' })` for older backends.

`MemorySqliteStoreRepository` is bound as the backend store. It applies migrations, keeps migration history in `pi_migrations`, enables WAL, migrates legacy `store.json` data without loss, and preserves the existing read/write/search/export/import-compatible `MemoryStoreRepository` shape. The JSON repository remains available as a compatibility fallback when `node:sqlite` is unavailable and as the backup/export shape.

## Data-preserving migration strategy

Memory migrations must be forward-only and data-preserving. Runtime migrations are the `MEMORY_SQLITE_MIGRATIONS` statements exposed through `MEMORY_SCHEMA`; whenever a table, column, index, trigger, or compatibility view is added, the shared schema and runtime migration list must stay aligned so `getSchema()` reflects the database that is actually created.

Existing user data is preserved through additive changes, backfills, and compatibility layers:

- New columns are introduced as nullable or with safe defaults, then backfilled from existing stable keys such as `workspace_key`, `workspace_path`, memory ids, repository locators, and payload metadata.
- Existing payload JSON remains the lossless source for fields that have not yet been normalized. Normalized tables such as knowledge concepts, links, memory relations, retrieval cache rows, workspace identity, and workspace id mappings are populated from that payload without deleting the original record.
- Renames or contract changes must use compatibility views, triggers, or dual-write/read translation until all callers have moved to the new shape. Destructive table rewrites, payload truncation, and implicit activation of imported or candidate data are not allowed.
- Workspace identity migrations must create or update `pi_workspaces` and backfill `workspace_id` into workspace-scoped tables without changing the existing `workspace_key` primary identity used by older records.
- Legacy `store.json` import into SQLite is an idempotent copy. The JSON file remains a backup/export/emergency fallback shape and must not be deleted or mutated by migration code.
- Sensitive content rules do not relax during migration. Backfills may copy ids, hashes, counts, redacted snippets, status, scope, timestamps, and evidence markers, but must not materialize raw prompts or secrets into new columns.

Each migration is applied once, recorded in `pi_migrations` with version, description, checksum, and timestamp, and should run inside the repository's SQLite transaction boundary. If a migration cannot complete, startup should leave the previous database contents readable through the last successful schema version or through the JSON fallback rather than partially rewriting user data. Recovery should prefer adding a corrective forward migration over editing an already released migration.

Validation for storage changes should include migration-history assertions, `MEMORY_SCHEMA` alignment with runtime tables, legacy `store.json` import coverage, and focused upgrade tests that seed old-shape records and verify that memories, events, graph snapshots, knowledge graph payloads, workspace settings, feedback, transcripts, skills, retrieval cache metadata, and benchmark reports survive the upgrade.

Documentation packages searched by Memory are global IDE assets from `~/.cybervinci/library/packages`. The same installed documentation index can serve any number of projects without copying or associating the package data with each workspace.

Workspace document and media ingestion is opt-in by format. `optIn.pdfDocuments` enables local `.pdf` extraction, `optIn.officeDocuments` enables local Office Open XML extraction for `.docx`, `.pptx`, and `.xlsx`, `optIn.images` enables local image metadata ingestion for `.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.bmp`, and `.ico`, and `optIn.diagrams` enables local diagram text ingestion for `.svg`, `.drawio`, `.mermaid`, `.mmd`, `.puml`, and `.plantuml`. These paths require files to pass ignore, denylist, size, and sensitive-path checks, run locally with best-effort extractors, and require no Python, container, service, paid API, model download, OCR, or network call. Image ingestion stores only safe metadata such as media type, size, and dimensions by default; it does not perform OCR. Diagram and document text is redacted before code/document chunks are persisted, and deterministic memory proposals from document/diagram content are stored only as review-required `candidate` memories with `source: pdf-document-ingestion`, `source: office-document-ingestion`, or `source: diagram-document-ingestion`. Remote image semantics are represented by `optIn.remoteImageSemantics` plus a recorded consent timestamp and must be configured explicitly by the application; the built-in local indexer does not call a remote provider.

## Optional Roslyn Extension

Roslyn is specific to the .NET compiler platform, mainly C# and Visual Basic. It is not part of the core Memory package. The core package exposes a backend analyzer contribution point and ships only the structural C# analyzer.

Install `@cybervinci/memory-roslyn` in an application that needs Roslyn-backed C# extraction. That package contains the TypeScript adapter, .NET sidecar source, environment variables, and smoke-test fixture. If the optional extension is not installed, Memory continues to index C# with the lightweight structural analyzer.

## Safety model

Memory is configurable by workspace. The UI exposes toggles for graph indexing, memory, skill suggestions, AI Chat learning, inline chat chips, auto-indexing, LLM chat learning, LLM learning frequency, optional learning model id, optional vector search, optional transcript search, optional PDF/Office/image/diagram ingestion, remote image semantics consent, ignore-rule handling, allowlist-only indexing, IDE/global memory export, and default retention policies for session memory, task memory, and transcripts. The default LLM learning frequency is `0`, which means no periodic schedule. Deterministic triggers can still ask an LLM for skill, memory, graph, and context suggestions when LLM chat learning is enabled. If LLM chat learning is disabled, the AI Chat integration remains deterministic-only. If no learning model id is configured, LLM learning uses the active AI Chat agent's default model. Vector search is off by default and requires an already configured local-only embedding backend; no model downloads are started silently. Transcript search is also off by default and is controlled by `optIn.transcriptSearch` in the Settings tab before transcript search can participate in retrieval. PDF, Office, image, and diagram ingestion are off by default and controlled by `optIn.pdfDocuments`, `optIn.officeDocuments`, `optIn.images`, and `optIn.diagrams`; extracted text is redacted before persistence and generated memory records remain review candidates. Workspace exports continue to omit IDE/global memories unless the caller or workspace setting explicitly enables `exportOptions.includeGlobalMemories`. `.gitignore` and `.cvignore`/`.cybervinciignore` handling is enabled by default and can be disabled from Settings for controlled indexing runs; denylist, allowlist, size, binary, generated, and secret checks remain local. Sensitive files and detected secrets are indexed as metadata only; raw secrets are not returned through findings. Context discovery is represented as suggestions and graph evidence, not automatic prompt injection.

Security filtering runs before chunking, embeddings, transcript persistence, audit logging, reports, and portable serialization. `.gitignore`, `.cvignore`, `.cybervinciignore`, denylist, allowlist-only mode, generated/binary/large-file checks, sensitive-path checks, and deterministic secret scanning produce structured ignore or redaction reasons that can be shown in the dashboard without copying ignored file content. Audit events store IDs, hashes, counts, scopes, statuses, and boolean evidence markers rather than raw secret-bearing text.

## Validation

Preferred package validation from the repository root:

```bash
npm run lint -- --scope @cybervinci/memory
npx lerna run --scope @cybervinci/memory test --stream --concurrency=1
```

Use narrower equivalent test commands when a change is limited to a single service, analyzer, repository, or UI behavior. Storage, migration, ranking, security, MCP, export/import, lifecycle, and portable artifact behavior should have focused unit coverage when changed.

Package-only TypeScript validation:

```bash
tsc -p packages/memory/tsconfig.json --noEmit
```

When validating from a worktree without dependencies, create a temporary junction to the original repo `node_modules`, run the command, and remove the junction before committing or handing off.
